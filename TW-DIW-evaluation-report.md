## 一、評估摘要（Executive Summary）

本報告評估數位發展部推動之「數位憑證皮夾（Taiwan Digital Identity Wallet, TW DIW）」於 Project SYNC — AI 驅動全人健康守護計畫中的技術適用性。

TW DIW 採用 W3C Verifiable Credentials（VC）與 Decentralized Identifiers（DID）國際標準，其「選擇性揭露」與「資料自主授權」的核心設計，與本專案「個人資料可攜性」的產品哲學高度契合。經評估，TW DIW 在行為處方箋的可驗證憑證化、社會處方箋的去中心化媒合、以及醫社數據閉環的信任基礎三大面向，均具備顯著的整合價值。

然而，考量 TW DIW 目前仍處於沙盒測試階段，健保卡整合尚在初期規劃，本報告建議採取「雙軌並行」策略：短期以健康存摺 SDK 為數據主幹，中期導入 TW DIW 作為跨機構信任層，以兼顧提案時程需求與長期架構願景。

| 技術標準成熟度 | ★★★★☆ | W3C VC / DID 已為國際標準，生態系持續擴大 |

## 二、TW DIW 技術概覽

### 2.1 定位與願景

數位憑證皮夾並非行動支付或電子錢包，而是一個專門用來安全儲存、展示與授權數位身分及各類憑證的國家級平台。由數位發展部主導開發，目標讓民眾透過手機統一管理政府核發（自然人憑證、健保卡、駕照）與民間發行（員工證、學生證、專業證照）的數位憑證。

### 2.2 核心技術架構

| 技術元素 | 標準 / 規範 | 功能說明 |
|----------|------------|---------|
| 可驗證憑證 | W3C Verifiable Credentials 2.0 | 以密碼學簽章確保憑證的真實性、完整性與不可篡改性 |
| 分散式識別符 | W3C Decentralized Identifiers | 去中心化身分識別，不依賴單一中央機構 |
| 選擇性揭露 | SD-JWT / BBS+ | 使用者僅揭露必要欄位，符合 GDPR 最少資訊揭露原則 |
| 本地儲存 | 裝置端加密 | 憑證儲存於手機本地，非雲端集中式資料庫 |
| 信任註冊表 | 公共區塊鏈 | 發行者公鑰與憑證狀態記錄於鏈上，供驗證者查詢 |
| 驗證方式 | QR Code / NFC | 支援掃碼與感應兩種出示方式 |

### 2.3 三方信任模型（Issuer-Holder-Verifier）

TW DIW 採用國際通行的三方信任模型：發行者（Issuer）簽發憑證、持有者（Holder）保管並自主決定出示內容、驗證者（Verifier）驗證憑證真偽。三方之間不需要即時連線或中央資料庫仲介，大幅降低資料外洩風險與系統耦合度。

### 2.4 推動時程

| 時間 | 里程碑 | 狀態 |
|------|--------|------|
| 2023 年 1 月 | 數位部加入 W3C，啟動 DID 應用研究 | 已完成 |
| 2024 年下半 | 系統開發與內部測試 | 已完成 |
| 2025 年 3 月 | 開放沙盒測試申請 | 進行中 |
| 2025 年底（目標） | 正式上線公共服務 | 延期評估中 |
| 2026 年 | 擴大民間應用場景、開源程式碼 | 規劃中 |

## 三、與 Project SYNC 的契合度分析

### 3.1 產品哲學的共鳴

Project SYNC 的產品哲學第四項「個人資料可攜性」——確保病患對自身健康數據擁有主權與自主授權能力——幾乎就是 TW DIW 的設計核心。此外，TW DIW 作為政府開放數據基礎建設，同時滿足了「政府開放資料」與「民間企業參與」兩項原則。

| Project SYNC 產品哲學 | TW DIW 對應能力 | 契合度 |
|----------------------|---------------|--------|
| 預防醫學出發 | 可作為行為處方箋的可信載體 | 間接支持 |
| 政府開放資料 | 國家級數位基礎設施，政府主導開發 | 直接對齊 |
| 民間企業參與 | 開放沙盒測試與未來開源，民間可開發自有皮夾 | 直接對齊 |
| 個人資料可攜性 | 資料自主授權 + 選擇性揭露 + 本地儲存 | 完全契合 |

### 3.2 三大整合場景

#### 場景一：行為處方箋的可驗證憑證化

醫師開立的「行為處方箋」可設計為一張 Verifiable Credential。在此模型中，醫師（或醫療機構）擔任 Issuer，病人擔任 Holder，社區據點或 AI 代理人擔任 Verifier。

病人到銀髮據點參加活動時，只需出示處方箋 VC 中的「活動類別」與「有效期間」，據點無需知道病人的完整病歷即可確認資格。這完美實踐了最少資訊揭露原則，同時為「數位藥方」提供了可信的技術底層。

#### 場景二：社會處方箋的去中心化媒合

自動化社會處方箋需要在病人、醫療機構、社區據點三方之間傳遞資料。TW DIW 的 Issuer-Holder-Verifier 三角架構天然適配此場景——病人握有自己的健康憑證，自主決定授權哪些資訊給哪個社區服務提供者，無需建置中央資料庫儲存所有人的交叉資料，降低資料外洩風險。

#### 場景三：醫社數據閉環的信任基礎

病人的社會參與紀錄（出席社區活動、完成微任務）若發行為 VC，回傳醫師後台時即具備密碼學可驗證性。這使得「健保論質計酬（P4P）」的依據不再只是 App 自行產生的 log，而是可供第三方獨立驗證的客觀數據，大幅提升計畫在健保署端的公信力。

## 五、風險與限制評估

### 5.1 健保卡整合仍在早期階段

健保署表示健保卡的數位皮夾整合目前僅規劃第一階段「顯示為憑證」。與醫療系統的深度串接（病歷存取、申報介接）因流程複雜仍需進一步討論。這意味著短期內 TW DIW 無法直接取代健康存摺 SDK 作為取得病患醫療數據的管道。

### 5.2 正式商用時程不確定

數位部原定 2025 年底前推出正式服務，但目前仍在沙盒測試階段。若 Project SYNC 要在 2026/04/03 前提交 Google.org 申請，TW DIW 可能尚未進入正式商用。在提案中應定位為「中期技術路線圖」而非「即可使用的基礎設施」。

### 5.3 醫療場域的特殊性

藥局領藥等場景因健保卡與後端醫療系統緊密耦合，整合難度較高。診間端的即時身分驗證，短期仍需仰賴既有的醫事憑證（HCA）體系。

### 5.4 使用者接受度

本專案主要服務對象為高齡慢性病患者，對新型數位工具的學習曲線較陡。TW DIW 的導入需搭配完善的 UX 行為設計與陪伴式引導，避免造成數位落差。

### 5.5 風險彙整

| 風險項目 | 影響程度 | 發生機率 | 緩解策略 |
|----------|---------|---------|---------|
| 健保卡整合延遲 | 高 | 高 | 短期採健康存摺 SDK，中期切換 |
| TW DIW 正式上線延期 | 中 | 中 | 提案中定位為中期路線圖 |
| 沙盒環境與正式環境差異 | 中 | 中 | 先行概念驗證，保留架構彈性 |
| 高齡使用者接受度低 | 高 | 中 | UX 行為設計 + 陪伴式引導 |
| VC 格式跨系統相容性 | 低 | 低 | 嚴格遵循 W3C 標準 |

## 六、建議導入策略：雙軌並行架構

綜合上述分析，建議 Project SYNC 採取「雙軌並行」的技術架構策略，以兼顧 Google.org 提案時程（2026/04/03 截止）與長期技術願景。

### 6.1 短期（MVP 階段）：健康存摺 SDK 為主幹

- **數據管道：** 健康存摺 SDK
- **身分驗證：** 自然人憑證 + 健保卡
- **資料交換：** App 內部資料庫 + API 對接

此階段利用現有成熟管道快速建立 MVP，確保能在 Google.org 提案中展示可運作的原型。

### 6.2 中期（規模化階段）：TW DIW 為信任層

- **導入項目：** W3C VC 格式的行為處方箋
- **導入項目：** W3C VC 格式的社會參與紀錄
- **導入項目：** Holder 自主授權取代中心化 API

此階段將 Project SYNC 的核心資料物件（處方箋、參與紀錄）標準化為 VC，透過 TW DIW 實現跨機構資料流通與病人自主授權，呼應 Google.org 重視的隱私保護與資料自主精神。

### 6.3 雙軌架構示意

| 層級 | 短期（MVP） | 中期（規模化） |
|------|------------|---------------|
| 身分驗證 | 自然人憑證 / 健保卡 | TW DIW + DID |
| 數據取用 | 健康存摺 SDK | 健康存摺 SDK + VC |
| 處方箋格式 | App 內部結構 | W3C Verifiable Credential |
| 資料授權 | OAuth / API Key | Holder 選擇性揭露 |
| 參與紀錄 | App log | 社區據點簽發 VC |
| P4P 佐證 | 系統匯出報表 | VC 附件（可驗證） |

## 八、結論

數位憑證皮夾（TW DIW）與 Project SYNC 在設計哲學上高度共鳴，其 W3C 標準基礎、選擇性揭露機制與去中心化架構，為本專案的行為處方箋、社會處方箋與醫社數據閉環提供了理想的技術底層。

儘管 TW DIW 目前仍處沙盒階段，健保場域的深度整合需要更長的政策推動週期，但透過「雙軌並行」策略，Project SYNC 可以在不犧牲短期交付能力的前提下，為中長期的跨機構信任架構預留升級路徑。

更重要的是，在 Google.org 提案中展示與國家數位基礎設施的政策對齊，以及對 W3C 國際標準與歐盟 eIDAS 2.0 趨勢的掌握，將顯著提升計畫的技術前瞻性與國際競爭力。

**建議立即啟動沙盒帳號申請與行為處方箋 VC 資料結構設計，作為提案書中技術準備度的有力佐證。**
